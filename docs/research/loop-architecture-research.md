# Autonomous Coding-Loop Architecture: Research Report and Design Implications for a Spec-Kit Extension

**Scope note on evidence:** the corpus below spans academic SWE-agent work (SWE-agent, OpenHands, SWE-bench, ReAct, Devin), the Ralph-loop practitioner literature (Huntley and derivatives), and the spec-kit / spec-driven-development documentation family. It contains **no controlled study of multi-agent vs. single-agent coding, no cross-model-judge benchmark, and no measured data on human-gate placement** — those sections below are triangulated from practitioner claims and paper tradeoffs, and gaps are flagged explicitly.

---

## 1. Taxonomy of coding-agent loop architectures

| Architecture | One-line mechanism | Source |
|---|---|---|
| **ReAct single-agent loop** | Interleave thought → action → observation; the plan is a live scratchpad continuously revised from environment feedback. | [ReAct: Synergizing Reasoning and Acting in Language Models](https://arxiv.org/abs/2210.03629) |
| **ACI-mediated ReAct (SWE-agent)** | Same loop, but the lever is the interface: purpose-built compact commands (search/view/edit-with-linter) replace the raw shell; a lint guardrail rejects bad edits inside every turn. | [SWE-agent](https://arxiv.org/abs/2405.15793) |
| **Event-stream platform agent (OpenHands)** | `step(state) → typed action` over a chronological event log, executed in a sandboxed per-session Docker runtime (bash + IPython + browser), with optional delegation to specialist agents. | [OpenHands](https://arxiv.org/abs/2407.16741) |
| **Long-horizon self-terminating agent (Devin)** | Plan → edit → run tests → observe → self-correct until pass, time-boxed (45 min) with self-termination; 72% of passing runs exceed 10 minutes. | [Cognition SWE-bench technical report](https://cognition.com/blog/swe-bench-technical-report) |
| **Ralph loop** | `while :; do cat PROMPT.md \| agent; done` — stateless fresh-context iterations, one task per loop, shared state via a committed plan file, correctness from external "backpressure" (compilers, tests) plus eventual consistency. | [Ralph Wiggum as a "software engineer"](https://ghuntley.com/ralph), [how-to-ralph-wiggum](https://github.com/ghuntley/how-to-ralph-wiggum) |
| **Spec-compile loop** | Spec (Markdown) is the source; a minimal reusable compile prompt regenerates code from it; humans edit the spec, never the code. | [Markdown as a programming language](https://github.blog/ai-and-ml/generative-ai/spec-driven-development-using-markdown-as-a-programming-language-when-building-with-ai), [Groundhog /specs](https://ghuntley.com/specs) |
| **Phased artifact pipeline (spec-kit)** | constitution → specify → clarify → plan → tasks → analyze → implement → converge; each phase emits a versioned Markdown artifact that gates and feeds the next; human checkpoints between phases. | [github/spec-kit](https://github.com/github/spec-kit), [Spec Kit docs](https://github.github.com/spec-kit) |
| **Architect/editor split (Aider)** | A strong reasoner emits plain-text change instructions; a second, format-reliable model translates them into applyable diffs — reasoning and mechanical edit application are separate calls. | [Aider edit formats](https://aider.chat/docs/more/edit-formats.html) |
| **Orchestrated agent fleet ("Gas Town")** | An orchestrator decomposes work into granular units (MEOWs) picked up by many ephemeral parallel agents — "Kubernetes for agents." Described, not demonstrated. | [Dev Interrupted podcast](https://devinterrupted.substack.com/p/inventing-the-ralph-wiggum-loop-creator) |

The operator's baseline pipeline is essentially spec-kit's phased pipeline with a Ralph-style build loop embedded at the Build step — which is exactly the fusion the evidence supports, provided the seams between the two are respected (see §6).

## 2. The verification/gating spectrum

Ordered weakest to strongest by what the evidence shows:

**Self-grading (Ralph-style agent self-assessment) — demonstrably unreliable.** The repomirror field report is blunt: "agent claimed 100% completion while some demos didn't actually work," output plateaued at ~90% and humans closed the gap in interactive sessions ([repomirror](https://github.com/repomirrorhq/repomirror/blob/main/repomirror.md)). Huntley documents the failure mode's mechanism: models "chase compiling code as reward and cheat with placeholder stubs unless explicitly forbidden" ([Ralph](https://ghuntley.com/ralph)). Spec-kit's own Phase -1 gates are "checklist self-assessments by the same agent, not independent verification — weaker than cross-model judging" ([spec-driven.md](https://github.com/github/spec-kit/blob/main/spec-driven.md)). Conclusion: never let the maker declare itself done.

**Generator-critic within the harness — cheap and measurably effective.** SWE-agent's edit command runs a linter before applying and forces retry on failure; ablations show removing this guardrail measurably degrades resolution rate ([SWE-agent](https://arxiv.org/abs/2405.15793)). This is a per-edit mechanical critic, not an LLM — the lesson is that the cheapest critic (syntax check, typecheck) placed *inside* every action beats deferred review.

**Cross-model / LLM-judge — used in practice, evidence is thin and honest about noise.** The Ralph playbook uses "LLM-as-judge with binary pass/fail for subjective criteria (tone, UX)" and explicitly concedes the judge "is itself non-deterministic (same artifact may pass/fail on different runs) — stated as an intentional tradeoff" ([how-to-ralph-wiggum](https://github.com/ghuntley/how-to-ralph-wiggum)). SWE-agent's pass@6 with a reviewer/selector improves over pass@1 ([SWE-agent](https://arxiv.org/abs/2405.15793)) — a selector-judge over multiple candidates has measurable value. **Gap flagged:** no source in this corpus benchmarks *cross-family* judging specifically; the independence argument for it is design reasoning (avoiding shared blind spots of same-family self-review), not measured evidence here.

**Tests-as-oracle — the strongest signal in the corpus, with known holes.** SWE-bench's core contribution is execution-grounded verification: "test-based verification gives an objective, executable correctness signal, enabling automated gating" ([SWE-bench](https://arxiv.org/abs/2310.06770)). Devin quantifies its power: making ground-truth fail-to-pass tests visible as an explicit target **nearly doubled** success, 13.86% → 23% ([Cognition](https://cognition.com/blog/swe-bench-technical-report)). Huntley's whole thesis is that "generating code is now cheap; verification/backpressure is the bottleneck," and he chose Rust specifically for compiler strictness ([Ralph](https://ghuntley.com/ralph)). But the oracle leaks: SWE-bench's test-pass "can be gamed or under-specified... passing the PR's tests is not proof the fix matches intent" ([SWE-bench](https://arxiv.org/abs/2310.06770)); SWE-agent observed "superficial fixes that pass reproduction but not hidden tests" ([SWE-agent](https://arxiv.org/abs/2405.15793)); and compile-gating alone yields code "baked with unspecified latent behaviours" ([Groundhog](https://ghuntley.com/specs)).

**Where self-correction fails, concretely:** placeholder-stub reward hacking ([Ralph](https://ghuntley.com/ralph)); false "not implemented" conclusions from non-deterministic search ([Ralph](https://ghuntley.com/ralph)); over-literal instruction following that faithfully implements a wrong suggestion ([Cognition](https://cognition.com/blog/swe-bench-technical-report)); looping on failed edits ([SWE-agent](https://arxiv.org/abs/2405.15793)); and inflated self-reported completion ([repomirror](https://github.com/repomirrorhq/repomirror/blob/main/repomirror.md)).

**Synthesis:** the working stack is layered — mechanical critic per edit, tests as the primary gate, an *independent* judge for what tests can't express, and a human for intent. No single layer suffices; every source that relies on one layer documents its escape routes.

## 3. Long-loop survival

**Termination/convergence.** Three proven mechanisms: (a) turn/cost budgets ([SWE-agent](https://arxiv.org/abs/2405.15793)); (b) time-box + self-termination — Devin's 45-minute cap with early exit ([Cognition](https://cognition.com/blog/swe-bench-technical-report)); (c) plan-file exhaustion — Ralph terminates when `fix_plan.md` empties, and a separate planning loop diffs code against specs to regenerate it ([Ralph](https://ghuntley.com/ralph)). Spec-kit's `/converge` is the same idea formalized: diff codebase vs. spec, append remaining work as delta tasks ([github/spec-kit](https://github.com/github/spec-kit)). One emergent curiosity: a repomirror agent self-terminated via `pkill` after detecting it was looping — do not rely on that ([repomirror](https://github.com/repomirrorhq/repomirror/blob/main/repomirror.md)).

**Context/memory management.** The numbers are consistent across independent sources: usable context degrades well before the advertised limit — Huntley cites ~147–152k of 200k ([Ralph](https://ghuntley.com/ralph)); the playbook targets **40–60% utilization as the "smart zone"** with one task per loop ([how-to-ralph-wiggum](https://github.com/ghuntley/how-to-ralph-wiggum)). Mechanisms: deterministic re-loading of the same core files every iteration (consistency over token cost) ([Ralph](https://ghuntley.com/ralph)); SWE-agent keeps only the last ~5 observations in full, collapsing older ones ([SWE-agent](https://arxiv.org/abs/2405.15793)); the main context acts as a *scheduler* that fans reads out to subagents (~156kb each, garbage-collected) but uses **only one** subagent for build/test to avoid backpressure collapse ([how-to-ralph-wiggum](https://github.com/ghuntley/how-to-ralph-wiggum)); spec-kit offloads detail to `implementation-details/` to keep plan.md navigable ([spec-driven.md](https://github.com/github/spec-kit/blob/main/spec-driven.md)). Counterintuitive but replicated: **shorter prompts win** — expanding from ~103 to ~1,500 words made agents "slower and dumber" ([repomirror](https://github.com/repomirrorhq/repomirror/blob/main/repomirror.md)).

**Cost control.** Concrete anchors: ~$10.50/hour per Sonnet loop, ~$800 for 6 overnight ports ([repomirror](https://github.com/repomirrorhq/repomirror/blob/main/repomirror.md)); full SWE-bench eval ~$6.9k ([OpenHands](https://arxiv.org/abs/2407.16741)); Huntley's $297-for-a-$50k-MVP anecdote is explicitly unverified ([Ralph](https://ghuntley.com/ralph)). Cost scales with wall-clock, so termination criteria *are* the cost control.

**Human-in-the-loop placement — the literature splits into two camps.** Spec-kit places a human checkpoint between *every* phase, on the argument that gaps compound: catch them in artifacts before implementation ([GitHub Blog](https://github.blog/ai-and-ml/generative-ai/spec-driven-development-with-ai-get-started-with-a-new-open-source-toolkit)). Ralph places the human *around* the loop, not in it: "watch the loop" and permanently engineer away each failure domain; a pause gate (Ctrl+C to advance) still counts as the same loop ([everything is a ralph loop](https://ghuntley.com/loop)). Both camps agree the human is not optional: "100%-autonomous-no-engineer claims are horseshit; expect ~90% done" ([Ralph](https://ghuntley.com/ralph)).

**Runaway prevention.** Git as checkpoint — commit+push+tag on passing tests, recovery = `git reset --hard` ([Ralph](https://ghuntley.com/ralph)); commit after *every* edit so progress survives crashes ([repomirror](https://github.com/repomirrorhq/repomirror/blob/main/repomirror.md)); sandbox as the sole security boundary when running `--dangerously-skip-permissions` ("not if it gets popped, but when") ([how-to-ralph-wiggum](https://github.com/ghuntley/how-to-ralph-wiggum)); per-session Docker isolation ([OpenHands](https://arxiv.org/abs/2407.16741)); prompt "signs" appended when misbehavior is observed ([Ralph](https://ghuntley.com/ralph)).

## 4. Multi-agent vs. single-agent

**The honest headline: this corpus contains no measurement showing multi-agent orchestration beats a single agent for coding.** What it contains:

- **Strong practitioner argument against:** Huntley explicitly rejects multi-agent multiplexing as premature complexity; "agents-as-microservices with non-deterministic components is a red hot mess" — the single-agent one-task-per-loop constraint exists *specifically* to avoid compounding non-determinism across agents ([everything is a ralph loop](https://ghuntley.com/loop)).
- **Measured near-parity:** OpenHands' delegation from CodeActAgent to a specialist BrowsingAgent scores 15.3% on WebArena vs. 15.5% for the specialist alone — delegation added machinery, not performance ([OpenHands](https://arxiv.org/abs/2407.16741)).
- **Where role-splitting demonstrably earns its cost — within one loop, not across a fleet:** (a) Aider's architect/editor split pairs a strong reasoner with a format-reliable editor at the price of one extra call ([Aider edit formats](https://aider.chat/docs/more/edit-formats.html)); (b) Ralph's scheduler + read-subagents pattern uses subagents as *memory extension*, not as peers ([how-to-ralph-wiggum](https://github.com/ghuntley/how-to-ralph-wiggum)); (c) SWE-agent's best-of-n with a reviewer/selector improves over pass@1 ([SWE-agent](https://arxiv.org/abs/2405.15793)).
- **Embarrassingly-parallel scale-out works when tasks are independent:** repomirror ran 6 concurrent loops as tmux sessions — but each was a full independent Ralph loop on its own repo, with zero inter-agent coordination ([repomirror](https://github.com/repomirrorhq/repomirror/blob/main/repomirror.md)). The moment coordination is needed, "coordination, not code generation, is the bottleneck" ([Dev Interrupted](https://devinterrupted.substack.com/p/inventing-the-ralph-wiggum-loop-creator)).

**Opinionated reading:** multi-agent helps when the second agent plays a *different role* (editor, judge, selector, read-scout) or works on a *fully independent task*. It adds cost and non-determinism when peer agents share mutable state on one task. For the operator's design this validates maker + judge (different roles) and invalidates any temptation toward swarms.

## 5. Harness/orchestration mechanics

**Checkpointing = git, and it must be automatic.** Two proven disciplines: commit-and-push after every file edit so any iteration crash loses nothing ([repomirror](https://github.com/repomirrorhq/repomirror/blob/main/repomirror.md)), or commit+tag only on green tests, making every commit a known-good restore point ([Ralph](https://ghuntley.com/ralph)). Recovery is `git reset --hard` vs. rescue prompts — a judgment call, but only possible if commits exist ([Ralph](https://ghuntley.com/ralph)). The operator's failure (E: fully hand-cranked, no auto-commit) is exactly the anti-pattern both sources engineer against.

**Durable state lives on disk, not in the session.** The Ralph pattern's core insight: iterations are stateless; `IMPLEMENTATION_PLAN.md` (committed) is the shared state; "the loop is dumb, the plan file is smart" ([how-to-ralph-wiggum](https://github.com/ghuntley/how-to-ralph-wiggum)). Spec-kit generalizes this to artifact-per-phase handoff: each phase's Markdown output is the structured context for the next ([Spec Kit docs](https://github.github.com/spec-kit)). This directly addresses the operator's failure (C): decisions must be written to files *as a mandatory step of each iteration*, because file state is the only thing the next iteration (or the Capture phase) can see. Ralph even has agents update `AGENT.md` as they learn and embed intent comments in tests as "notes to future fresh-context iterations" ([Ralph](https://ghuntley.com/ralph)).

**Subagents vs. separate sessions.** Within an iteration: subagents for expensive/context-polluting reads (many, parallel, disposable), exactly one for build/test ([how-to-ralph-wiggum](https://github.com/ghuntley/how-to-ralph-wiggum)). Across iterations: fresh sessions with deterministic file loading — "deterministic stack allocation," wasteful but consistent ([Ralph](https://ghuntley.com/ralph)). The judge should be a separate session/model entirely, since same-agent checklist self-assessment is the documented weak point ([spec-driven.md](https://github.com/github/spec-kit/blob/main/spec-driven.md)).

**Workflow engines: the evidence says stay minimal.** An agent is "~300 lines of code running in a loop plus tokens" ([everything is a ralph loop](https://ghuntley.com/loop)); OpenHands' abstraction is one function, `step(state) → action` ([OpenHands](https://arxiv.org/abs/2407.16741)); spec-kit's orchestration is slash commands + prerequisite-checking helper scripts + templates, with layered template resolution for customization ([github/spec-kit](https://github.com/github/spec-kit), [Microsoft Dev Blog](https://developer.microsoft.com/blog/spec-driven-development-spec-kit)). SWE-agent's deeper point applies to the harness itself: interface details (guardrails, compact feedback, output suppression) change agent success more than orchestration sophistication does ([SWE-agent](https://arxiv.org/abs/2405.15793)).

## 6. Design implications for the extension

### 6.1 Phase the loop: split Build from everything downstream — this fixes failure (A)

The over-scoped iteration violated the single most-replicated finding in this corpus: **one task per loop, 40–60% context utilization** ([Ralph](https://ghuntley.com/ralph), [how-to-ralph-wiggum](https://github.com/ghuntley/how-to-ralph-wiggum)). Build/e2e/verify/ship/sign-off in one iteration budget guarantees the late steps get starved — which is precisely what happened. Restructure as:

- **Inner Build loop (Ralph-style, per task):** pick one task from tasks.md → implement → run scoped tests/typecheck (the SWE-agent per-edit critic, cheap and in-loop [SWE-agent](https://arxiv.org/abs/2405.15793)) → write decision record → commit → exit iteration. Loop until tasks.md is exhausted or budget/time-box hits (Devin's 45-min pattern [Cognition](https://cognition.com/blog/swe-bench-technical-report)).
- **Outer pipeline (spec-kit-style, per feature):** Frame → Plan → Analyze → [Build loop] → Review → Verify → Ship → Capture, each a *separate session* consuming the previous phase's artifacts ([Spec Kit docs](https://github.github.com/spec-kit)). Review and Verify are not iterations of the Build loop; they are phases that the Build loop cannot reach into or skip.

### 6.2 Gate placement — direct answer: gate per phase-*boundary*, not one human accept at the end, and not a human per iteration

The two camps in the evidence resolve cleanly when you notice they gate different granularities. Spec-kit gates **phases** with humans because artifact errors compound downstream ([GitHub Blog](https://github.blog/ai-and-ml/generative-ai/spec-driven-development-with-ai-get-started-with-a-new-open-source-toolkit)); Ralph runs **iterations** unattended because per-iteration human review destroys autonomy ([everything is a ralph loop](https://ghuntley.com/loop)). Adopt both:

- **Human STOP #1 — after Plan/Analyze, before Build.** This is where spec-kit's evidence is strongest: "catching gaps before they compound into implementation" ([GitHub Blog](https://github.blog/ai-and-ml/generative-ai/spec-driven-development-with-ai-get-started-with-a-new-open-source-toolkit)), and Huntley's costliest failure was upstream — "a single duplicated keyword in a lexer spec cost ~1 month; spec quality dominates outcomes" ([Ralph](https://ghuntley.com/ralph)). A cheap human read of the plan is the highest-leverage minutes in the whole pipeline.
- **No human inside Build → Verify.** Build iterations are gated automatically (tests, typecheck, judge). Running build→verify unattended is exactly what the Ralph/repomirror evidence shows works ([repomirror](https://github.com/repomirrorhq/repomirror/blob/main/repomirror.md)).
- **Human STOP #2 — after Verify, before Ship.** This is non-negotiable on the evidence: agent self-assessment overstates completion ([repomirror](https://github.com/repomirrorhq/repomirror/blob/main/repomirror.md)), test oracles are gameable ([SWE-bench](https://arxiv.org/abs/2310.06770)), and expect "~90% done" ([Ralph](https://ghuntley.com/ralph)). Ship is irreversible-ish; the human signs here, once, with the full Review+Verify evidence in front of them.

So: **not** "build→verify→ship then a single accept" (ship must sit *behind* the accept), and **not** a gate on every phase (Frame→Plan can flow; Build iterations must flow). Two human STOPs, everything else automated.

- **Fixing failure (B) — make Review structurally unskippable:** implement Review (/code-review + Semgrep + /security-review) as its own phase whose *output artifact is a prerequisite* for Verify, using spec-kit's prerequisite-checking mechanic (implement refuses to run unless constitution+spec+plan+tasks exist — same pattern, new artifact) ([github/spec-kit](https://github.com/github/spec-kit), [Microsoft Dev Blog](https://developer.microsoft.com/blog/spec-driven-development-spec-kit)). The conflation happened because Review lived inside the loop's checker; the fix is a hard file-existence gate the harness checks, not the agent.

### 6.3 What to automate vs. keep manual

**Automate (evidence-backed):**
- **Auto-commit, immediately** — commit after every edit or at minimum on every green iteration; this is table stakes in every long-loop source and fixes failure (E) ([repomirror](https://github.com/repomirrorhq/repomirror/blob/main/repomirror.md), [Ralph](https://ghuntley.com/ralph)). Wire it as a Claude Code hook so the agent cannot forget it.
- **Decision recording as a per-iteration hard step** — fixes failure (C). Make "append decision to decisions file" part of the iteration's exit checklist, enforced the way Ralph enforces plan-file updates before commit ([how-to-ralph-wiggum](https://github.com/ghuntley/how-to-ralph-wiggum)); Capture then reads guaranteed-populated files.
- **Contract-text reconciliation on accepted deviation** — fixes failure (D). Spec-driven doctrine is explicit: the spec is the mutable source of truth; "to change course, edit the spec, regenerate" ([GitHub Blog](https://github.blog/ai-and-ml/generative-ai/spec-driven-development-with-ai-get-started-with-a-new-open-source-toolkit)), and spec-kit's `/converge` exists precisely to re-diff code vs. spec and reconcile ([github/spec-kit](https://github.com/github/spec-kit)). Rule: accepting a deviation *is* a spec edit — the accept action must trigger a contract-update task before the loop continues, plus an ADR (mirroring spec-kit's Complexity Tracking written-justification pattern [spec-driven.md](https://github.com/github/spec-kit/blob/main/spec-driven.md)).
- **In-loop mechanical backpressure** — typecheck/lint on every edit ([SWE-agent](https://arxiv.org/abs/2405.15793)), scoped tests per changed unit ([Ralph](https://ghuntley.com/ralph)).

**Keep manual:** plan approval (STOP #1), ship approval (STOP #2), deviation acceptance, and "watching the loop" — treating each observed misbehavior as a permanent harness fix (prompt sign, new gate) rather than a one-off correction ([everything is a ralph loop](https://ghuntley.com/loop)). Also keep runtime/browser smoke-testing manual until instrumented: spec-kit flags that runtime errors are invisible to artifact gates ([github/spec-kit](https://github.com/github/spec-kit)).

### 6.4 Model topology: local maker + cross-family judge — supported, with two caveats

The intended topology maps onto three evidence-backed patterns: (1) **role-split beats peer-swarm** — architect/editor shows heterogeneous models paired by role is the multi-model pattern that pays ([Aider edit formats](https://aider.chat/docs/more/edit-formats.html)); (2) **independent selection improves outcomes** — pass@6 + reviewer beats pass@1 ([SWE-agent](https://arxiv.org/abs/2405.15793)); (3) **same-model self-checking is the documented weak layer** ([spec-driven.md](https://github.com/github/spec-kit/blob/main/spec-driven.md), [repomirror](https://github.com/repomirrorhq/repomirror/blob/main/repomirror.md)) — an independent judge is the structural fix, and a cross-family judge maximizes that independence. Also precedented: Huntley already offloads to a different model family when the primary hits limits ([Ralph](https://ghuntley.com/ralph)).

Two caveats the evidence forces:

- **Weak local makers need a stronger harness, not hope.** SWE-agent's gains "don't fully transfer to small local models without capable instruction-following" ([SWE-agent](https://arxiv.org/abs/2405.15793)), and Aider shows weak models fail exact-match diff formats — select edit format per model, falling back to whole-file for the local maker ([Aider edit formats](https://aider.chat/docs/more/edit-formats.html)). Compensate with tighter task decomposition (small isolated units, "almost TDD for your agent" [GitHub Blog](https://github.blog/ai-and-ml/generative-ai/spec-driven-development-with-ai-get-started-with-a-new-open-source-toolkit)) and maximal mechanical backpressure.
- **The judge gates *subjective* criteria only; tests remain the primary oracle.** Devin's 13.86%→23% jump from visible target tests ([Cognition](https://cognition.com/blog/swe-bench-technical-report)) argues for writing failing acceptance tests at Plan time (spec-kit's confirmed-failing-Red-phase, test-first mandate [spec-driven.md](https://github.com/github/spec-kit/blob/main/spec-driven.md)) and letting the loop chase them. The cross-family judge sits on top with **binary pass/fail** verdicts ([how-to-ralph-wiggum](https://github.com/ghuntley/how-to-ralph-wiggum)) — and because judge verdicts are themselves non-deterministic (same source), a judge PASS should be advisory into Review, while a judge FAIL is a hard block back into the loop. **Flagged gap:** no source here quantifies cross-family vs. same-family judging; this recommendation rests on the documented failure of self-assessment plus independence-by-design reasoning, not a benchmark.

### 6.5 Summary blueprint

Frame → Plan → Analyze → **[HUMAN STOP: approve plan + failing acceptance tests]** → Build loop (local maker, one task/iteration, 40–60% context, per-edit lint/typecheck, per-iteration decision-record + auto-commit, cross-family judge as iteration exit gate, budget/time-box termination) → Review phase (own artifact, prerequisite-enforced) → Verify (full test suite + judge over the whole diff; accepted deviations force contract-update + ADR before pass) → **[HUMAN STOP: accept]** → Ship → Capture (reads the now-guaranteed decision files). Each of the operator's five failures maps to one structural fix above; none requires more than the mechanics already demonstrated in the cited sources.
